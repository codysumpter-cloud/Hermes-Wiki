### 3.1 `96af61b6e` `feat(memory,skills): approve/deny gate for memory + skill writes (#38199)`

`2026-06-09T21:51 -0700`，1270+/11− 跨 13 文件。

**问题源**：用户反馈 background self-improvement review fork 经常 unprompted 保存 "wrong assumption"。

**初始三态 `write_mode`**（`on | off | approve`），applied to 前台 turn 和后台 self-improvement review fork：
- `on`（默认）：自由写，行为不变
- `off`：永不写；tool 返回 clean disabled result
- `approve`：不 commit。Memory 前台写**内联**提示（small，chat bubble 可 review）；background memory 写 + **所有** skill 写**暂存**到 pending store（SKILL.md 太大不能 inline review，daemon thread 不能 block 在 prompt 上）

**从 CLI 或任何 messaging 平台 review 暂存写入**：

```
/memory pending|approve|reject|mode
/skills pending|approve|reject|diff|mode
```

Skill review 尊重大小不对称：inline 看到 one-line gist；完整 unified diff 走 out-of-band（`/skills diff`、dashboard、staged JSON）。

**新文件**：`tools/write_approval.py`（**491 行**初版） + `hermes_cli/write_approval_commands.py`（197 行）— 共享 CLI+gateway handler。Gate 接入单一入口 `memory_tool()` 和 `skill_manage()`，借助既有 write-origin `ContextVar` 区分 foreground 和 `background_review`。

## 16. Params / 错误分类 — newer OpenAI 系列 max_completion_tokens 名匹配 + 不误判 unsupported-param 为 overflow

### 16.1 `19c07c403` `fix(params): send max_completion_tokens for newer OpenAI families on custom endpoints`

第三方 OpenAI-compatible endpoint（self-hosted gateway / OpenRouter / Azure proxy）前置 gpt-4o / gpt-4.1 / gpt-5+ / o1-o4 model 静默接收 `max_tokens` 然后 400 with `unsupported_parameter`，因为三个 kwarg-selection site 只检查 `base_url_hostname(...) == "api.openai.com"` 而 fall through 到 `max_tokens` 在任何其他 host 上。这个约束**由 model family 在服务器端强制**，不是 URL，所以名字检测作为 fallback **必需**。

**修改**：
- `utils.py`：新共享 helper `model_forces_max_completion_tokens(model)` — 前缀匹配 `gpt-4o / gpt-4.1 / gpt-5 / o1 / o3 / o4` family 在 normalized（lowercased、vendor-prefix-stripped）name 上
- `run_agent.py`：`_max_tokens_param` OR 入 helper 检查
- `agent/auxiliary_client.py`：`auxiliary_max_tokens_param` 加可选 keyword-only `model` 参数；`_build_call_kwargs` inline 分支 apply 同样检查给 `provider == "custom"` 和 non-custom 路径

测试：`tests/test_model_forces_max_completion_tokens.py` **31 新 case**（正 family、负（classic gpt-4 / claude / llama / mistral / qwen / deepseek）、vendor prefix、大小写、空白、None/empty、substring-not-prefix 守护）+ `tests/run_agent/test_run_agent.py::TestMaxTokensParam` 5 个新 model-based 测试。

### 16.2 `2ce3ae3d1` `fix(error-classifier): don't misclassify unsupported-param 400s as context overflow`

GPT-5 model reject `max_tokens` → 400 message 包含字面 `max_tokens` 字符串 — 一个 `_CONTEXT_OVERFLOW_PATTERNS`。`_classify_400` 在任何 request-validation 检查（只存在于 5xx 路径）**之前**检查 overflow pattern → 参数错误被路由进 compression loop，用同样 bad param re-send，以 'Cannot compress further' 在微小 context 上结束。

**修复**：在 `_classify_400` 里把一个 request-validation 守护（unsupported/unknown parameter）hoist 到 context-overflow 检查之上。故意排除通用 `invalid_request_error` code，OpenAI 在真实 overflow 400 上也盖那个章 — 所以真 overflow 仍 compress。配合 `max_completion_tokens` 参数 fix 在源头 stop bad request。

## 33. Install / Update / Backup — Windows winget 陈旧注册 + PATH 合并 + 中断 venv 自愈 + backup SQLite 同位放置 + 嵌套 skill 目录纳入

### 33.1 `899acfe42` `fix(install/windows): repair stale winget registration; refresh PATH after every package manager`

`ripgrep/ffmpeg` missing 时 `winget install <id>` 在 winget 已注册的 package 上**当 upgrade 处理**：没 newer version → exit `0x8A15002B` (`APPINSTALLER_CLI_ERROR_UPDATE_NOT_APPLICABLE`)，**无视 binary 是否实际存在**。Installer 只 log 那个 code，且按 `Get-Command rg` 判 success，所以 stale registration（在 winget 外删除文件 / 缺失 alias shim）变成**永久 dead-end** — winget 一直报 "already installed" 用户**永远无法重装**。

检测那个 exit code 并 retry 一次 `--force` 修复 registration → shim 重出现。

也在 `choco` 和 `scoop` fallback 之后**刷新** process PATH（不仅是 winget）通过共享 helper → 成功 fallback install（或在无 winget 的盒上的任何 install）不再被误报 "not installed"。
