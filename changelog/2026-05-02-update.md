## 重大新功能

### 1. Persistent Goals (`/goal`) — Ralph 循环

**源码**：`hermes_cli/goals.py:1-535`、`cli.py:7046-7140`、`gateway/run.py:7648-7790`

跨 turn 持久化目标：用户 `/goal <text>` 设置，每 turn 结束后辅助模型评判，未完成则自动续写。

**核心设计不变量**（goals.py:13-28）：

- continuation prompt 是普通 user message，**不修改系统提示，不动 toolset**——prompt cache 完整保留
- judge 失败 fail-OPEN：返回 `continue`，turn budget 是 backstop（默认 20 turns）
- 真实用户消息抢占 continuation——但完成后仍重新评判
- 模块对 `cli.HermesCLI` 和 gateway 零硬依赖——双写入点

**状态机**（`GoalState`，goals.py:89-119）：

```
active → done           # judge 判定满足
active → paused         # turn 预算耗尽 / 用户 /goal pause
paused → active         # /goal resume（默认重置 budget）
* → cleared             # /goal clear（保留审计记录）
```

**持久化**：`SessionDB.state_meta` 表，key `goal:<session_id>`（goals.py:127-204）。`/resume` 自动捡回。

**Judge prompt**（goals.py:61-81）固定：返回 `{"done": bool, "reason": "..."}` JSON。提取容错：strip markdown fence + regex 抽 first JSON object（goals.py:223-265）。

**判决三态**：
- `done`：goal 标记完成，停止
- `continue`：发 continuation prompt，turns_used += 1
- `skipped`：辅助模型不可达——fail-open `continue`（goals.py:289-302）

**调用链**：

| 入口 | 触发位置 | 行 |
|------|---------|-----|
| CLI `/goal` | `_handle_goal_command` | cli.py:7082 |
| Gateway `/goal` | `_handle_goal_command` | gateway/run.py:7680 |
| 后置评判 | `evaluate_after_turn` | goals.py:441 |

详见 [[persistent-goals-ralph-loop]]。

### 2. Tool-call Loop Guardrails

**源码**：`agent/tool_guardrails.py:1-456`、`run_agent.py:165-167, 1160, 1653-1657, 9152-9188`

防止模型陷入"重复调用同一失败工具"或"重复读取同一只读结果"循环。配置在 `tool_loop_guardrails`，默认 **warnings 启用，hard stop 关闭**——CLI/TUI 用户得到温和提示，需要 circuit breaker 时显式开启。

**核心数据**（tool_guardrails.py:62-80）：

```python
@dataclass(frozen=True)
class ToolCallGuardrailConfig:
    warnings_enabled: bool = True
    hard_stop_enabled: bool = False
    exact_failure_warn_after: int = 2
    exact_failure_block_after: int = 5
    same_tool_failure_warn_after: int = 3
    same_tool_failure_halt_after: int = 8
    no_progress_warn_after: int = 2
    no_progress_block_after: int = 5
```

**三大检测维度**：

1. **exact failure**（同 `tool_name` + 完全相同 args 反复失败）—— 用 `ToolCallSignature.from_call` 算 args 的 SHA256（tool_guardrails.py:127-140）
2. **same-tool failure**（不论 args，同 tool 名连续失败）
3. **idempotent no progress**（read-only tool 重复返回相同结果）—— `IDEMPOTENT_TOOL_NAMES` frozenset 包含 `read_file`、`search_files`、`web_search`、`web_extract`、`session_search`、`browser_snapshot/console/get_images`、`mcp_filesystem_*`（tool_guardrails.py:19-38）

**互斥分类**（tool_guardrails.py:377-380）：`MUTATING_TOOL_NAMES` 包含 `terminal/execute_code/write_file/patch/todo/memory/skill_manage/browser_click/type/press/scroll/navigate/send_message/cronjob/delegate_task/process` —— 这些工具永不被视为 idempotent。

**决策动作**（`ToolGuardrailDecision.action`，tool_guardrails.py:143-172）：`allow` / `warn` / `block` / `halt`。
- `warn`：通过 `append_toolguard_guidance` 在 tool result 末尾追加 `[Tool loop warning: ...]`
- `block` / `halt`：通过 `toolguard_synthetic_result` 生成合成 JSON `{"error": ..., "guardrail": ...}`，并在 `_set_tool_guardrail_halt` 中记录终止决策

**与 `agent.display._detect_tool_failure` 一致**：`classify_tool_failure`（tool_guardrails.py:188-218）显式镜像 CLI 的 `[error]` 标签判定，避免 guardrail 与可见错误标签出现分歧。

**集成路径**（run_agent.py）：

- `__init__` 实例化 controller（行 1160），配置 `from_mapping` 解析 `agent.tool_loop_guardrails`（行 1653-1657）
- `_append_guardrail_observation`（行 9166）—— 每次 `after_call` 后追加观测
- `_guardrail_block_result`（行 9186）—— 在 `before_call` 拦截时返回合成结果
- `_set_tool_guardrail_halt` 维护 first-halt 决策，提供 `_toolguard_controlled_halt_response` 给最终 assistant 回复

详见 [[tool-loop-guardrails]]。

### 3. Kanban — durable multi-profile collaboration board

**源码**：`hermes_cli/kanban_db.py:1-2765`、`hermes_cli/kanban.py:1-1393`、`tools/kanban_tools.py:1-727`、`plugins/kanban/dashboard/plugin_api.py`

跨 profile 共享的 SQLite 任务面板，路径固定 `$HERMES_HOME/kanban.db`（**profile-agnostic on purpose**——同机所有 profile 看到同一面板，这就是协作原语）。

**架构基石**（kanban_db.py:1-18）：

- WAL 模式 + `BEGIN IMMEDIATE` 写事务 + CAS 在 `tasks.status` 和 `tasks.claim_lock`
- SQLite 自身串行化 writers——losers 看到 0 affected rows 直接走开，**没有重试循环、没有分布式锁机器**
- `workspace_kind` 字段把协作和 git worktree 解耦，让 research/ops/digital-twin workload 与 coding workload 共存

**任务状态机**（VALID_STATUSES，kanban_db.py:38）：
```
triage → todo → ready → running → done|blocked → archived
```

**三个 workspace 种类**（VALID_WORKSPACE_KINDS，kanban_db.py:39）：`scratch` / `worktree` / `dir`

**Claim TTL**：默认 15 分钟（kanban_db.py:45）。worker 通过 `heartbeat_claim` 保活；超期后下一次 dispatcher tick 自动回收。

**CLI 表面**（15-verb，kanban.py:156-419）：`init / create / list(ls) / show / assign / link / unlink / claim / comment / complete / block / unblock / archive / tail / dispatch / daemon(deprecated) / watch / stats / notify-{subscribe,list,unsubscribe} / log / runs / heartbeat / assignees / context / gc`

**Worker tools**（tools/kanban_tools.py，仅在环境变量 `HERMES_KANBAN_TASK` 存在时注册）：

```
kanban_show / kanban_complete / kanban_block / kanban_heartbeat /
kanban_comment / kanban_create / kanban_link
```

**为什么用 tools 而不是 shell**（kanban_tools.py:8-25）：

1. **Backend portability**：worker 跑在 Docker/Modal/Singularity/SSH 时，`hermes` 可能不在容器里；tools 直接走 Python 进程访问 `~/.hermes/kanban.db`
2. **No shell-quoting footguns**：`--metadata '{"x":[...]}'` 透 shlex+argparse 易碎；结构化 args 跳过
3. **Better errors**：tool-call 失败返回结构化 JSON 而非 stderr

**Dispatcher 嵌入 gateway**（kanban.py:299-314）：旧的 `hermes kanban daemon` 已废弃但保留——通过隐藏 `--force` flag。设计意图：避免双 dispatcher 旧坑。

**Dashboard 插件**（plugins/kanban/dashboard/plugin_api.py）：FastAPI APIRouter 挂在 `/api/plugins/kanban/`，写路径与 CLI/gateway 共享。**WebSocket 用 query token 鉴权**（浏览器无法在 upgrade 请求设 Authorization 头）。

**Worker context caps**（kanban_db.py:53-57）：避免病态 board 把 LLM prompt 撑爆——每个 task body 8KB，每个 comment 2KB，最多 30 个最近 comments、10 个 prior attempts。

详见 [[kanban-multi-profile-board]]。
